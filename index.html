Why It Matters: Binary search is the canonical O(log n) technique. Every FAANG interview expects it on sorted arrays, rotated arrays, and — critically — on the answer space itself (binary search on value, not index).

Sliding Window
Core DSA Technique
45 problems · subarray, substring, frequency maps, deque variants
Why It Matters: Converts O(n²) brute-force subarray enumeration into O(n) single-pass solutions by maintaining a live window [left, right] updated incrementally. The "aha" of recognizing when the window should shrink is a genuine test of algorithmic intuition.
Core Idea: The window always represents a valid/optimal range. Instead of recomputing from scratch, add one element at right and remove one at left. The invariant is what makes the window valid — if you can't define it, the technique doesn't apply.

Kadane's Algorithm
Algorithms Technique · DP
28 problems · max subarray, circular, product, deletion, 2D, tree variants
Why It Matters: Kadane's Algorithm is a linear-time DP technique that finds the maximum-sum contiguous subarray by maintaining one invariant: at each index i, compute the best subarray sum ending exactly at i — either extend the previous best or start fresh. The recurrence maxEndingHere = max(nums[i], maxEndingHere + nums[i]) is deceptively simple, yet it underpins problems spanning circular arrays, products, 2D matrices, absolute values, and DP-on-tree variants. It appears consistently in top-tier interviews because (a) naive O(n²) is trivially wrong, (b) the optimal is O(n), and (c) extensions like circular arrays and product subarrays test whether a candidate truly understands the invariant or has merely memorised the code.
Core Invariant: maxEndingHere = max(nums[i], maxEndingHere + nums[i]) — start fresh when maxEndingHere < 0 because a negative prefix can never help. Global answer = max over all maxEndingHere values.

State: dp[i] = max subarray sum ending at i  ·  Recurrence: dp[i] = max(nums[i], dp[i−1] + nums[i])  ·  Preferred: Bottom-up, O(1) space with rolling variables (maxEndingHere, maxSoFar). Top-down adds call-stack overhead for zero benefit.

🚩 2D problem without column-compression — Running Kadane's row-by-row misses cross-row rectangles. Fix column boundaries → compress rows → apply 1D Kadane's.

Merge Intervals
Algorithms Greedy · Sweep · DP · Segment Tree
37 problems · merge, insert, sweep, coverage, weighted DP, online CRUD, 2-D, circular
Why It Matters: Merge Intervals models the fundamental computer science primitive of managing contiguous ranges on a linear axis — time slots, memory addresses, genomic base pairs, network port ranges, log timestamps. The core invariant is devastatingly simple: after sorting by start, two closed intervals [a,b] and [c,d] overlap iff c ≤ b. Every single variant in this topic — from classic merge to sweep-line to segment trees — is a direct consequence of this one inequality. It appears at Google, Meta, Stripe, and Jane Street because it tests greedy reasoning, proof justification, and the ability to escalate a simple idea into an online/streaming/2-D context under pressure.
The Mental Model
Think of intervals as paint strokes on a number line. Merging is asking: after all strokes dry, what are the distinct painted segments? Every sub-variant reduces to this image:
• Classic merge → what are the final painted segments?
• Count overlaps → at which point is the paint thickest?
• Insert interval → add one new stroke, re-report the segments
• Remove overlaps → erase minimum strokes so no two wet strokes touch
• Gap detection → where are the unpainted gaps?
• Weighted scheduling → which strokes to keep to maximise total value?

When stuck on any interval problem, return to this image and ask: "What am I computing about this paint?"
Core Invariant & Proof Sketch
Invariant: After sorting intervals by start time, for any two consecutive intervals Ii = [ai, bi] and Ii+1 = [ai+1, bi+1], they overlap iff ai+1 ≤ bi.

Proof sketch (why sorting by start is sufficient): Suppose intervals are sorted: a1 ≤ a2 ≤ ... ≤ an. If Ii and Ij overlap with j > i+1, then aj ≤ bi. But ai+1 ≤ aj ≤ bi, so Ii and Ii+1 also overlap. Therefore, any non-consecutive overlap is caught by a chain of consecutive overlaps.

Consequence: We never need to look back more than one step — the greedy single-pass is complete.
Corollary: The merged result is unique (order of merging does not matter), which justifies the greedy approach's correctness without needing an exchange argument.

🔧 #16 — Circular Interval Merge: Given a circle with M units (0 to M−1) and N intervals [L_i, R_i]; an interval is circular if L_i > R_i (wraps around). 🔧 #32 — Point-Stabbing Count Queries: Given N intervals [L_i, R_i] and Q query points p_j, return for each query the count of intervals containing p_j. When This Technique BREAKS
🚩 Intervals have weights and you want maximum total value — Greedy merge selects wrong intervals. Use Weighted Job Scheduling DP (binary search + DP sorted by end-time).
🚩 Intervals arrive online with O(log n) per-op requirement — Static sort-and-merge is O(n) rebuild per insert. Use TreeMap / SortedList for O(log n) predecessor/successor lookup.
🚩 Axis is huge (up to 10⁹) and you need a difference array — Plain array of size 10⁹ causes MLE. Use coordinate compression or a Fenwick tree on compressed coordinates.
🚩 Circular/modular intervals — Sort-by-start fails at wrap-around. Split circular intervals at 0/M boundary, run linear merge, then restitch.
🚩 Intervals are 2-D (rectangles) — 1-D merge gives wrong area. Use sweep-line on x-axis + 1-D interval merge/segment tree on y-axis.
🚩 Containment detection (is A fully inside B?) — Standard overlap check c ≤ b catches containment AND partial overlap — you need the stricter check a ≤ c AND d ≤ b. Classic merge logic silently gives wrong answer here.
🚩 "Burst Balloons" style problems — The intervals represent DP sub-problems, not ranges to merge. Use Interval DP (dp[i][j]), not sweep-line.
🚩 Streaming with hard memory limits — Storing all intervals for offline sort is impossible. Use reservoir-style online merge or accept approximate answers with a bounded-error sketch.
Proof of Correctness Skeleton
1. Classic Greedy Merge (#1)
Claim: The sort + single-pass algorithm produces the unique minimum set of disjoint intervals covering the same union.
Invariant: After processing the first k intervals (sorted by start), result contains the correct merged intervals for those k intervals.
Inductive step: At interval k+1 with start s and end e: if s > result.last.end, no previously processed interval reaches s (since all starts ≤ s by sort), so k+1 is disjoint → append. If s ≤ result.last.end, they overlap; max(result.last.end, e) is the correct merged end because no future interval (start ≥ s) can "fill" a gap inside [result.last.start, s−1].
Uniqueness: The merged output is unique because the union set is fixed and the greedy produces the coarsest partition.
2. Activity Selection / Non-overlapping Removal (#3)
Claim: Greedy "keep the interval with earliest end-time" maximises the number of kept non-overlapping intervals.
Exchange argument: Let G = greedy solution, O = any optimal, both sorted by end-time. If they agree on the first k intervals but differ at k+1 — G picks g, O picks o with end(g) ≤ end(o) (greedy choice). Swap o for g in O: g doesn't overlap with O's (k+2)-th interval (since end(g) ≤ end(o) and O was valid). Count doesn't decrease.
Conclusion: G can be transformed into O without reducing count → G is optimal.
3. Weighted Job Scheduling DP (#17)
Claim: dp[i] = max(dp[i−1], profit[i] + dp[p(i)]) where p(i) = latest job ending before start[i] gives maximum profit.
Observation: For job i, either it is NOT included (best = dp[i−1]) or it IS included (best = profit[i] + best using only jobs ending before start[i] = dp[p(i)]). These two cases are exhaustive and disjoint, so dp[i] is correct.
Complexity: p(i) found via binary search on end-times in O(log n), giving O(n log n) total.

Red Flags / Common Mistakes
🚩 Applying greedy merge to weighted intervals — Classic merge has no notion of value — it merges all overlapping intervals regardless. If any interval carries a weight and you need to SELECT a subset, you must use Weighted Job Scheduling DP. This is the single most common wrong approach at the Hard level.
🚩 Confusing "overlap" with "containment" — [1,10] and [3,5] overlap AND contain each other. Problems asking to remove covered intervals (LC 1288) require checking a ≤ c AND d ≤ b, not just c ≤ b. Using the overlap check here silently over-counts removals.
🚩 Forgetting the heap pop in Meeting Rooms II — Not popping before pushing means you never reuse a freed room, inflating the answer. Always check heap[0] ≤ cur.start before deciding to push vs. heapreplace.
🚩 Online vs offline conflation — Collecting all intervals and sorting at the end is wrong for streaming problems (My Calendar, Range Module). It breaks the online contract and will fail any follow-up that asks "what if intervals arrive in real time?"
🚩 Difference array with values up to 10⁹ — Allocating diff = [0] * (10**9 + 1) causes MLE. Coordinate-compress the endpoints first, then build the difference array on compressed indices only.
🚩 Not handling the empty input or single-interval edge cases — Many interval bugs surface only when len(intervals) == 0 or == 1. Always handle these before the loop.
🚩 Circular interval wraparound — Treating [22, 3] on a 24-hour clock as a normal interval with start > end causes incorrect sort ordering and false non-overlap conclusions. Always split first.

Prefix Sums
Algorithms Pre-computation · Hash Map · Difference Array · 2D · BIT
30 problems · 5 stages · range queries, difference arrays, prefix XOR, 2D, dynamic BIT
Why It Matters: Prefix Sums is the foundational pre-computation primitive of competitive programming. The core invariant is: sum(i, j) = prefix[j+1] − prefix[i] — any contiguous subarray sum reduces to a single O(1) subtraction after O(n) preprocessing. This single identity unlocks an enormous class of problems: range sum queries, subarray count problems, 2D rectangle queries, difference arrays for range updates, and hash-map-augmented subarray problems. It appears in virtually every Google/Meta interview round because it tests whether a candidate instinctively pre-computes rather than recomputes — a fundamental signal of algorithmic maturity. At the ICPC/IOI level, prefix sums compose with segment trees, sparse tables, and Fenwick trees to form the backbone of nearly all range query algorithms.
The Mental Model
A prefix sum array is a "running water level" — prefix[i] is the total water accumulated up to position i. The water between two banks i and j is simply the level at j minus the level at i. You measure the river once; you answer every query in one subtraction.

How the mental model scales through the 5 stages:
• Stage 1: prefix[j+1] − prefix[i] = sum of arr[i..j]. One array, one subtraction.
• Stage 2: The "water" is not integers but modular residues, XORs, or parities. The subtraction becomes a modular operation or XOR — same shape, different algebra.
• Stage 3: The river becomes a 2D lake. prefix[r][c] = total water in the rectangle from (0,0) to (r,c). Inclusion-exclusion replaces the single subtraction.
• Stage 4: The problem never says "subarray sum." You must recognise that a condition like sum(i,j) = k is secretly prefix[j] − prefix[i] = k → prefix[i] = prefix[j] − k. The water level is hidden inside a counting or existence question.
• Stage 5: There are 10⁹ positions but only 10⁵ updates. The river is sparse. A Fenwick tree or segment tree replaces the static array — the identity prefix[j] − prefix[i] still holds, but each level query is now O(log n).
Core Invariant & Proof Sketch
Invariant: For a 1-indexed array a, define prefix[0] = 0 and prefix[i] = a[1] + a[2] + ... + a[i]. Then for any 0 ≤ i ≤ j ≤ n:
    sum(i+1, j) = prefix[j] − prefix[i]

Proof sketch (by telescoping): By definition, prefix[j] = a[1] + ... + a[j] and prefix[i] = a[1] + ... + a[i]. Subtracting: prefix[j] − prefix[i] = a[i+1] + ... + a[j] — the shared prefix a[1..i] cancels exactly.

Generalisation: This telescoping works for any associative operation with an inverse: addition (subtraction), XOR (XOR again), multiplication (division if non-zero). It does NOT work for max/min because there is no inverse — this is the fundamental limitation.

2D corollary: rect(r1,c1,r2,c2) = prefix[r2][c2] − prefix[r1−1][c2] − prefix[r2][c1−1] + prefix[r1−1][c1−1] by inclusion-exclusion on the same telescoping identity applied twice.

Custom Problem Statements
🔧 #13 — 2D Range Add, Point Query: Given an N×M grid initialised to all zeros, process Q operations. Each operation is (r1, c1, r2, c2, v): add v to every cell (r, c) where r1 ≤ r ≤ r2 and c1 ≤ c ≤ c2. After all operations, answer P point queries: what is the value at cell (r, c)?
Constraints: 1 ≤ N, M ≤ 1000, 1 ≤ Q, P ≤ 10⁵, 1 ≤ v ≤ 100.
Example: N=3, M=3; ops=[(1,1,2,2,5), (2,2,3,3,3)]; queries=[(1,1),(2,2),(3,3)] → [5, 8, 3].
Intended solution: Maintain 2D difference array diff[N+2][M+2]. For each op (r1,c1,r2,c2,v):
  diff[r1][c1] += v; diff[r1][c2+1] −= v; diff[r2+1][c1] −= v; diff[r2+1][c2+1] += v
After all ops, compute 2D prefix sum of diff to recover the grid. O(NM + Q) total.
Pattern Recognition Triggers
Linguistic Triggers:
• "sum of elements between index i and j" → Stage 1 — direct range query; build prefix array
• "number of subarrays with sum equal to k" → Stage 2 — prefix[j] − prefix[i] = k; hash map on prefix values
• "add v to all elements in range [l,r], then query point" → Stage 3 — difference array (inverse prefix)
• "submatrix sum" or "rectangle sum" → Stage 3 — 2D prefix with inclusion-exclusion
• "count subarrays where [some property of sum holds]" → Stage 4 — transform property into prefix condition; check what algebra applies
Structural Triggers:
• Input is an array + multiple (l, r) range queries with no updates → Stage 1/2, static prefix array
• Input is an array + multiple (l, r, v) range updates + point queries → Stage 3, difference array
• Input is a 2D grid + rectangle queries or updates → Stage 3, 2D prefix or 2D difference array
Constraint Triggers:
• N ≤ 10⁵ with Q ≤ 10⁵ queries AND updates → Stage 5; static prefix rebuild is O(n) per update → use Fenwick tree
• Values up to 10⁹ and N ≤ 10⁵ with no updates → Stage 1–4; static prefix is fine
Anti-Triggers (when NOT to use prefix sums):
🚩 "maximum subarray" or "maximum of subarray [i..j]" → NOT prefix sum (max has no inverse) → Kadane's algorithm or sparse table (static) or segment tree (dynamic)
🚩 "number of subarrays with sum ≥ k" → NOT hash-map prefix (inequality ≠ equality) → Monotone deque on prefix (LC 862) or binary search + BIT
🚩 "sum of all subsets" → NOT prefix sum; it's a bitmask/combinatorics problem counting each element's contribution across 2ⁿ subsets

Q1: Does the problem ask about a contiguous range (subarray, submatrix, path segment) of an array/grid? → NO: Not a prefix sum problem. Stop. → YES: Continue. Q2: Are there UPDATES to array values between queries? → YES: Go to Q8 (dynamic prefix — Stage 5). → NO: Continue. Q3: Is the query asking for RANGE SUM (or equivalent: XOR, product, count of property)? → NO: Go to Q7. → YES: Continue. Q4: Is the answer a single range sum value? → YES: Stage 1 — build prefix array, answer = prefix[j+1] − prefix[i]. Q5: Is the condition an EQUALITY (sum = k, sum % k = 0, XOR = k)? → YES: Stage 2 — hash map on prefix values. Key: transform "sum(i,j) = k" → "prefix[j] − k exists in map". Expected complexity: O(n). → NO: Go to Q6. Q6: Is the condition an INEQUALITY (sum ≥ k, sum in [lo, hi])? → YES: ⚠️ Hash map BREAKS here. When This Technique BREAKS
🚩 Range MAX or MIN queries (no inverse exists) — max(i..j) cannot be expressed as f(prefix[j]) − f(prefix[i]) because max has no inverse operation. [Stage 1–2 mistake] → Sparse Table (O(1), static) or Segment Tree (O(log n), dynamic).
🚩 Inequality conditions on subarray sums (≥ k, ≤ k, ∈ [lo,hi]) — Hash map lookup requires exact equality. [Stage 2 mistake] → Monotone Deque for contiguous window variants; Merge Sort on Prefix Array / BIT with coordinate compression for arbitrary ranges.
🚩 Dynamic updates to the array (online, streaming) — Rebuilding the entire prefix array after each update is O(n) per update = O(nq) total. [Stage 5 mistake] → Fenwick Tree (BIT) for point updates + prefix queries; Segment Tree for range updates + range queries.
🚩 Non-associative operations (e.g. GCD, LCM of a range) — GCD has no simple inverse (gcd(a,b) − gcd(a,c) ≠ gcd(b,c)). [Stage 1 mistake] → Sparse Table for static GCD/LCM range queries in O(1).
🚩 2D difference array with overlapping boundaries — A common Stage 3 mistake is applying 1D logic to 2D — forgetting the +v at (r2+1, c2+1) (inclusion-exclusion term). [Stage 3 mistake] → Always derive the 2D formula from first principles: four corners, alternating signs.
🚩 Circular/wrap-around subarrays — prefix[j] − prefix[i] breaks when the subarray wraps around index 0. [Stage 2 mistake] → Handle as total_sum − (non-wrapping complement), or double the array and apply standard prefix on arr + arr.
🚩 Very large axis (up to 10⁹) with sparse updates — A static array of size 10⁹ causes MLE. [Stage 5 mistake] → Coordinate-compress endpoints to O(q) distinct values, then apply difference array or BIT on compressed indices.
Proof of Correctness Skeleton
Stage 1 — 1D Range Sum (Telescoping Invariant)
Claim: prefix[j] − prefix[i] = a[i+1] + ... + a[j] for all 0 ≤ i ≤ j.
Proof by definition and telescoping: Define prefix[0] = 0; prefix[k] = prefix[k−1] + a[k] for k ≥ 1. Then prefix[j] − prefix[i] = (a[1]+...+a[j]) − (a[1]+...+a[i]) = a[i+1]+...+a[j]. □
Note: Correctness is unconditional — no edge cases beyond index bounds.
Stage 2 — Subarray Sum Equals K (Hash Map Correctness)
Claim: The running hash-map algorithm counts exactly the number of pairs (i,j) with i < j and sum(i+1..j) = k.
Proof by invariant maintenance:
• Invariant: at step j, freq[v] = number of indices i ≤ j with prefix[i] = v.
• At each step, we query freq[prefix[j] − k] = number of i < j with prefix[i] = prefix[j] − k = number of valid left endpoints for subarrays ending at j.
• We initialise freq[0] = 1 (empty prefix at i=0), ensuring subarrays starting from index 1 are counted.
• We update freq[prefix[j]] AFTER querying, preventing i = j from being counted. □
Stage 3 — Difference Array (Inverse Correctness)
Claim: After applying all (l, r, v) operations to diff, computing prefix_sum(diff) recovers the final array value at every index.
Proof by linearity: Each operation contributes v to every index in [l, r]. Encoding as diff[l] += v, diff[r+1] −= v means prefix_sum(diff)[i] = Σ v over all operations where l ≤ i ≤ r. This holds because for operation (l, r, v), prefix sum accumulates v starting at l (where it was added) and stops at r+1 (where it was subtracted). The contribution to index i is v iff l ≤ i ≤ r. □

Red Flags / Common Mistakes
🚩 [Stage 1] Initialising prefix array of size n instead of n+1 — Without the sentinel prefix[0] = 0, you cannot represent the subarray starting at index 0. Every range query is off by one. Correct model: "prefix[i] = sum of first i elements; prefix[0] = 0 always."
🚩 [Stage 2] Using hash map for inequality conditions (≥ k, ≤ k) — Hash maps only resolve exact equality. Correct model: "Inequality requires a sorted structure — deque, BIT, or merge sort."
🚩 [Stage 2] Not initialising freq[0] = 1 before the loop — This misses subarrays that start from index 0. A subarray [0..j] has sum = prefix[j] − 0 = prefix[j]. Correct model: "The empty prefix (before index 0) is a valid left boundary."
🚩 [Stage 3] Confusing difference array direction — Applying diff[l]+=v, diff[r]-=v (forgetting the +1 on r) when the interval is inclusive [l,r] subtracts one position too early, corrupting index r. Correct model: "Subtract at the position AFTER the endpoint — prefix sum must carry the value through r."
🚩 [Stage 4] Trying to apply prefix sums to MAX/MIN queries — A candidate drilled on prefix sums will attempt prefix_max[j] − prefix_max[i] for range max. Algebraically unsound. Correct model: "Only operations with an inverse (+, XOR, ×) support the prefix trick. Max/min have no inverse."
🚩 [Stage 5] Rebuilding static prefix after each update (O(n) per update) — For n=10⁵ and q=10⁵ updates, this is 10¹⁰ ops. Correct model: "Any update invalidates all subsequent prefix values — you need a data structure designed for dynamic prefixes."
🚩 [Stage 3] Computing 2D prefix sum in wrong order (column before row) — Building prefix[i][j] requires prefix[i-1][j], prefix[i][j-1], prefix[i-1][j-1] all finalised. Correct model: "Always iterate rows outer, columns inner — or verify all dependencies are filled before use."

Backtracking
Algorithms Recursion · Constraint Search · Pruning · Bitmask · DLX
30 problems · 5 stages · subsets, permutations, grid, CSP, memoised backtracking
Why It Matters: Backtracking is systematic constrained search with undo — the universal algorithm for problems where you must explore a decision tree, commit to choices one at a time, and retract them when they violate constraints or exhaust possibilities. The core invariant is: at every node in the decision tree, the partial solution is valid; when a dead end is reached, exactly one decision is undone and the next alternative is tried. It appears in every top-tier interview because it tests recursive reasoning, state management, pruning intuition, and the ability to convert a combinatorial description into executable code. At ICPC/IOI, backtracking with aggressive pruning (branch-and-bound, constraint propagation) solves NP-complete problems like SAT, graph colouring, and exact cover — appearing in competitions disguised as scheduling, placement, or configuration problems.
The Mental Model
Backtracking is building a sentence word by word. At each position you try every valid word, write it down, move to the next position, and erase it when you get stuck — trying the next word instead. You never erase more than the last word you wrote.

How the mental model scales through the 5 stages:
• Stage 1: You choose from a fixed set at each position; every choice is independent. The "sentence" is a permutation or combination.
• Stage 2: One constraint flips — elements can repeat, or you can't reuse the same position, or order doesn't matter. The choice set shrinks by one rule.
• Stage 3: The state itself becomes 2-dimensional — you're filling a grid, a board, or making choices across two linked structures simultaneously.
• Stage 4: The problem doesn't say "enumerate" or "find all." It says "is it possible?" or "find minimum cuts." You must recognise that the search space IS a backtracking tree, hidden inside a constraint satisfaction or optimisation problem.
• Stage 5: The naive tree is too wide — exponential branches at every node. Pruning transforms it: constraint propagation eliminates branches before exploring them, memoisation collapses subtrees, or bitmask DP replaces the explicit tree.
Core Invariant & Proof Sketch
Invariant: At every recursive call backtrack(state, choices), the partial state satisfies all constraints imposed so far, and choices is exactly the set of valid extensions of state.

Soundness: Every solution added to the result satisfies all constraints because we only add when the base case is reached AND all constraints have been checked at each decision point. No invalid partial state is ever extended.

Completeness: Every valid solution is found because the algorithm explores ALL branches of the decision tree — it never skips a valid extension. Pruning only eliminates subtrees that can be proven to contain zero valid solutions.

Termination: The recursion depth is bounded by the solution length (at most n), and at each level the choice set strictly decreases (each element is either used or the index advances). The tree is finite.

Undo correctness: The undo step restores the state to exactly what it was before the choice, because backtracking modifies state in exactly one way per call and reverts that exact modification. This is the key implementation invariant — state before = state after for any failed branch.

Custom Problem Statement
🔧 Bridge Problem — Backtracking with Constraint Propagation (Stage 4→5 bridge): You are given an N×N grid. Place exactly K non-attacking rooks such that no two rooks share a row or column, AND each rook must be placed in a white cell (cells are 0 = black or 1 = white). Return the count of valid placements.
Constraints: 1 ≤ N ≤ 8, 1 ≤ K ≤ N.
Example: N=3, K=2, grid=[[1,0,1],[0,1,0],[1,1,1]] → 6.
Intended solution: Backtrack row by row. At row r, try each column c where grid[r][c]==1 and col_used[c]==False. Mark col_used[c]=True, recurse to r+1, unmark. Base case: placed K rooks → count++. Pruning: if remaining_rows < K − placed, return (can't place enough). Bridges Stage 3 (N-Queens structure) into Stage 4 (recognising applicability in a new domain) with Stage 5 entry-level pruning.
Pattern Recognition Triggers
Linguistic Triggers:
• "generate all / enumerate all / find all possible" → Stage 1/2 — exhaustive enumeration of a decision tree
• "partition into groups / segments / subsets satisfying [constraint]" → Stage 3/4 — partitioning backtracking; each decision splits the input
• "place N pieces on a board such that no two [conflict]" → Stage 3 — placement backtracking with conflict sets
• "is it possible to [reach / form / achieve] using [limited resources]" → Stage 4 — feasibility search; decision tree over resource allocation
• "restore / reconstruct / decode all valid [strings/sequences]" → Stage 4 — reconstruction backtracking; validity check gates each branch
Structural Triggers:
• Output is List[List[...]] (all combinations, all partitions) → backtracking with path collection at base case
• Output is a single boolean ("is it possible?") with no polynomial algorithm obvious → backtracking with early-exit on first valid solution
• Input is a small N (≤ 20) with exponential search space but no DP state definition → backtracking with pruning
Constraint Triggers:
• N ≤ 20 with output as a count or existence → backtracking (possibly with bitmask DP if overlapping subproblems detected)
• N ≤ 9 for board-filling problems (Sudoku, N-Queens) → backtracking with constraint sets; O(9^N) feasible
Anti-Triggers (when NOT to use backtracking):
🚩 "minimum number of operations to reach target" → NOT backtracking (exponential) → BFS (unweighted) or Dijkstra (weighted)
🚩 "count the number of ways to [fill / partition]" where N > 20 → NOT plain backtracking → DP with memoisation; count is too large to enumerate
🚩 "find the shortest path / minimum cost path" → NOT backtracking → BFS / Dijkstra / Bellman-Ford; backtracking has no shortest-path guarantee
Decision Framework
Q1: Does the problem ask to ENUMERATE, CONSTRUCT, or CHECK EXISTENCE of solutions in a combinatorial space? → NO: Not a backtracking problem. Stop. → YES: Continue. Q2: Is N ≤ ~20 (making exponential search feasible)? → NO (N > 20): Backtracking will TLE. Go to Q9 (DP/BFS alternative). → YES: Continue. Q3: Does the problem ask for ALL solutions or ONE solution? → ALL: Full enumeration backtracking — collect at base case. → ONE: Backtracking with EARLY EXIT on first valid solution. When This Technique BREAKS
🚩 [Stage 1] N > 20 for enumeration problems — O(2²⁰) ≈ 10⁶ is feasible; O(2³⁰) ≈ 10⁹ is not. → Use bitmask DP if overlapping subproblems, or meet-in-the-middle (split into two halves of size n/2, enumerate each, combine).
🚩 [Stage 2] Counting problems with large N — Backtracking counts by enumerating every solution — O(result size), can be astronomical. Use DP where the count is accumulated without enumeration.
🚩 [Stage 3] Shortest-path or minimum-cost problems on a graph — Backtracking finds valid solutions but not minimum-cost ones efficiently. Use BFS (unweighted minimum steps) or Dijkstra (weighted).
🚩 [Stage 4] Polynomial solutions disguised as search — Some "NP-hard-looking" problems have greedy or DP solutions. Always check: optimal substructure (→ DP), exchange argument (→ greedy) before reaching for backtracking.
🚩 [Stage 5] Backtracking without memoisation on repeated states — If same (remaining, constraint-state) is reached via multiple paths, pure backtracking recomputes it. Convert to memoised recursion / bitmask DP. Classic symptom: TLE on N ≈ 15–20.
🚩 [Stage 3] Sudoku-style without constraint propagation — Bare backtracking Sudoku is O(9^81) worst case. Without "only try digits missing from row/col/box," TLE on hard puzzles. Forward checking is mandatory at this stage.
🚩 [Stage 5] Undo bugs making state corrupted — If undo is incomplete (global variable modified in one branch, not restored), subsequent branches see corrupted state. Symptom: wrong answers that are hard to reproduce. Every mutation in the choose step must be explicitly reversed in unchoose.
Proof of Correctness Skeleton
Stage 1 — Subset Enumeration (Invariant Maintenance)
Claim: backtrack(i, path) adds to result exactly all subsets of nums[i:] that extend path.
Proof by strong induction on n − i:
• Base case (i = n): No elements remain; path itself is the only valid extension → add path to result. ✓
• Inductive step: Assume correct for all calls with i' > i. At index i, we make two recursive calls: include — backtrack(i+1, path+[nums[i]]) correctly enumerates all subsets of nums[i+1:] extended by path+[nums[i]] by induction; exclude — backtrack(i+1, path) correctly enumerates all subsets of nums[i+1:] extended by path. Together they cover all subsets of nums[i:] extended by path. ✓
• Completeness: every include/exclude combination is tried. No duplicates: each path corresponds to a unique include/exclude sequence. □
Stage 3 — N-Queens (Constraint-Set Invariant)
Claim: The algorithm places exactly one queen per row such that no two queens share a column, diagonal, or anti-diagonal.
Proof by invariant maintenance:
• Invariant: Before placing queen in row r, sets cols, diag (r−c), anti_diag (r+c) contain exactly the columns/diagonals attacked by queens in rows 0..r−1.
• Placement: we only place at column c if c ∉ cols, (r−c) ∉ diag, (r+c) ∉ anti_diag — ensuring no conflict.
• Update: add c, r−c, r+c to the three sets, maintaining invariant for row r+1.
• Undo: after recursion, remove c, r−c, r+c, restoring invariant for trying next column in row r. ✓
• Completeness: all columns 0..n−1 are tried for each row. No valid placement is skipped. □
Stage 5 — Memoised Backtracking (Overlapping Subproblem Collapse)
Claim: Adding memoisation on (index, state) to backtracking produces the same result as without memoisation.
Proof by substitution: If backtrack(i, state) is called twice with identical arguments, both operate on identical remaining choices and constraints. By the Stage 1 inductive proof, both produce identical results. Therefore storing and returning the cached result is correct. ✓
Complexity improvement: Without memo, the same (i, state) subtree may be recomputed exponentially many times. With memo, each unique (i, state) is computed once → O(states · branching factor) total.

Red Flags / Common Mistakes
🚩 [Stage 1] Appending path reference instead of path copy — result.append(path) stores a reference to a list that will be mutated. All results become identical. Correct model: "I'm storing a snapshot — always path[:]."
🚩 [Stage 2] Skipping duplicates with i > 0 instead of i > start — Using i > 0 skips elements at the beginning of deeper recursion levels, silently dropping valid combinations. Correct model: "Skip a duplicate only if it's at the SAME level — i > start enforces same-level scope."
🚩 [Stage 3] Missing the undo step for grid problems — Marking visited=True but forgetting =False means cells remain blocked for sibling branches. Symptom: fewer results than expected. Correct model: "Every mutation before a recursive call must be reversed after it — no exceptions."
🚩 [Stage 4] Treating a counting problem as enumeration — "How many valid arrangements exist?" can sometimes be solved via DP without enumeration. Backtracking counts by generating every solution — exponential. Correct model: "If N > 15, check for DP before backtracking."
🚩 [Stage 4] Not recognising a game-theory problem as backtracking — "Can player 1 guarantee a win?" is a minimax tree. Candidates try BFS or greedy. Correct model: "Optimal play = exhaustive game-tree search. A position is winning iff any child is losing for the opponent."
🚩 [Stage 5] Running backtracking without pruning on constraint problems — Unconstrained Sudoku is O(9^81) — effectively infinite. Correct model: "Constraint propagation is mandatory, not optional. Always narrow the choice set before recursing."
🚩 [Stage 5] Confusing "backtracking with memo" with DP — When you memoise backtrack(index, state), you have written top-down DP. Not recognising this signals incomplete mastery. Correct model: "Memoised backtracking IS DP. Name it correctly and explain the state definition."

Greedy
Algorithms Exchange Argument · Sort Key · Heap · Regret · Matroid
32 problems · 5 stages · activity selection, heap greedy, regret, matroid, greedy + binary search
Why It Matters: Greedy algorithms make the locally optimal choice at each step — the proof that local optimality cascades into global optimality is an exchange argument: any optimal solution can be transformed into the greedy solution by swapping choices one at a time without worsening the result. Greedy is the most dangerous topic in competitive programming because it is simultaneously the easiest to apply and the easiest to apply wrongly — a greedy that looks correct on examples can fail on a single adversarial case. It appears in every Google interview because it tests mathematical reasoning (can you prove your greedy?), not just implementation. At ICPC/IOI, the hardest greedy problems require recognising that a problem with no obvious greedy structure reduces to a canonical greedy form (scheduling, matroid, exchange-argument) after a non-obvious transformation.
The Mental Model
Greedy is a vending machine that always dispenses the "best available" item. The key question is never "which item is best?" — it's "what does 'best' mean in this problem, and does picking it now guarantee we never regret it later?"

How the mental model scales through the 5 stages:
• Stage 1: "Best" is obvious — largest value, smallest cost, earliest deadline. Pick greedily in sorted order. No regret possible.
• Stage 2: "Best" changes based on one flipped constraint — earliest end instead of largest value, or smallest ratio instead of smallest absolute value. The sort key changes; the algorithm skeleton is identical.
• Stage 3: Two competing greedy criteria must be balanced simultaneously. The state tracks two quantities; the sort key is a composite function of both.
• Stage 4: The problem doesn't look like a greedy problem. You must first transform the input or reframe the objective before the "best available" principle becomes visible.
• Stage 5: The greedy choice requires knowing the current minimum/maximum of a dynamic set — a data structure (heap, segment tree, sorted set) makes each greedy step O(log n) instead of O(n).
Core Invariant & Proof Sketch — The Exchange Argument
Invariant: Let G = greedy solution, O = any optimal solution. We show G is at least as good as O.

3-step template applicable to ANY greedy problem:
• Step 1 (Align): Order both solutions by the greedy sort key. Find the first position k where G and O differ — G chose element g, O chose element o, with greedy_key(g) ≤ greedy_key(o) (G chose "better" by its criterion).
• Step 2 (Exchange): Swap o for g in O at position k. Show that after the swap, O' is still a valid solution AND cost(O') ≤ cost(O) (or value(O') ≥ value(O)). The entire proof lives here.
• Step 3 (Iterate): Repeat until O has been transformed into G. Since each swap didn't worsen O, cost(G) ≤ cost(O). Since O was optimal, G is also optimal. □

Why this matters: If you cannot execute Step 2, your greedy is wrong. The exchange argument IS the proof. Every greedy problem at Google requires you to sketch this argument when asked "how do you know this is correct?"

Custom Problem Statement (#29)
🔧 Weighted Matroid Greedy — Maximum Weight Spanning Forest (Stage 4): Given an undirected graph with N vertices and M weighted edges, find the maximum-weight spanning forest (a set of edges with no cycle, not necessarily connected, with maximum total weight).
Constraints: 2 ≤ N ≤ 10⁵, 1 ≤ M ≤ 3×10⁵, 1 ≤ weight ≤ 10⁹.
Example: N=4, M=5, edges=[(1,2,10), (2,3,6), (1,3,5), (3,4,4), (1,4,3)] → 20 (edges: (1,2,10) + (2,3,6) + (3,4,4)).
Intended solution: Sort edges by weight descending. Greedily add each edge if it doesn't form a cycle (Union-Find check). This is Kruskal's algorithm for maximum spanning forest. Why greedy works: the set of acyclic subgraphs of a graph forms a graphic matroid — and greedy always finds the maximum-weight independent set in a matroid. Exchange argument: any two spanning forests of the same size can exchange edges one at a time without losing the forest property.
Pattern Recognition Triggers
Linguistic Triggers:
• "minimum number of [operations / jumps / arrows] to [achieve]" → Stage 1/2 greedy — activity selection or reach-extension; local minimum choice is never regretted
• "maximum number of [events / tasks / intervals] you can [attend / complete]" → Stage 2/3 — activity selection sorted by end time; heap for dynamic scheduling
• "arrange / order / sequence to [maximise / minimise] total" → Stage 4 — sort key derived from exchange argument; comparator a·B vs b·A or a+b vs b+a
• "at each step, you can [action]; find minimum steps" → Stage 2 — prefix-reach greedy; reach = max(reach, f(i))
• "assign [resources] to [tasks] optimally" → Stage 3/5 — heap-based greedy; assign best-available resource at each step
Structural Triggers:
• Input is a list of items each with two attributes (value + weight, start + end, cost_A + cost_B) → sort by a function of both attributes; exchange argument determines the function
• Output is a single number (min count, max count) NOT a specific arrangement → likely greedy with proof, not backtracking
• Problem has a "never regret" property visible in examples (choosing X now never prevents a better choice later) → greedy applicable
Constraint Triggers:
• N ≤ 10⁵ with no obvious DP state → greedy is the intended solution; DP states would require O(n²) or more
• Items have two attributes and you must select ALL items (not a subset) in an optimal order → custom comparator sort; exchange argument for sort key
Anti-Triggers (when NOT to use greedy):
🚩 "maximum value subset of weight ≤ W" → NOT greedy (fractional knapsack is greedy, 0/1 knapsack is NOT) → use 0/1 Knapsack DP
🚩 "minimum cost path from source to destination" → NOT greedy scan → Dijkstra (non-negative) / Bellman-Ford (negative)
🚩 "number of ways to [achieve target]" → NOT greedy (greedy gives one way, not a count) → DP with counting recurrence
Decision Framework
Q1: Does the problem ask for an OPTIMAL (min/max) outcome achievable by making sequential choices? → NO: Not a greedy problem. Stop. → YES: Continue. Q2: Does the problem have OPTIMAL SUBSTRUCTURE without OVERLAPPING SUBPROBLEMS (i.e., choices don't interact with each other)? → NO (choices interact): → Use DP. → YES (choices are independent once sorted): Continue. Q3: Can you identify a SORT KEY such that processing in that order never creates regret (a later element is NEVER hurt by an earlier choice)? → YES (obvious: end time, ratio, etc.): → Stage 1/2. Sort, then linear scan. Expected: O(n log n). → MAYBE (sort key requires exchange argument): → Stage 4. Derive comparator: "does choosing a before b improve result?" Q4: Does the greedy choice depend on the CURRENT BEST AVAILABLE item in a DYNAMIC collection (items arrive over time or are unlocked)? → YES: → Stage 3/5 — use a HEAP. Min-heap if smallest-cost; max-heap if largest-value. Expected: O(n log n). → NO: Continue. Q5: Does making a greedy choice sometimes turn out to be WRONG and need to be UNDONE based on future information? → YES: → Stage 4 — REGRET GREEDY. Make the greedy choice; push an "undo token" to a heap; if a better option arrives, swap via the heap. Examples: Course Schedule III, Furthest Building, Min Refueling Stops. → NO: Continue. Q6: Is the problem a STRING ORDERING or ARRANGEMENT problem? → YES: → Stage 4 — EXCHANGE ARGUMENT COMPARATOR. For two items a, b, ask: "Is a+b better than b+a?" Derive the comparator from this comparison. Sort by custom comparator. Expected: O(n log n). → NO: Continue. Q7: Is the greedy check INSIDE a binary search on the answer? → YES: → Stage 5 — GREEDY + BINARY SEARCH. Binary search on threshold/count; greedy verifies feasibility. Expected: O(n log² n) or O(n log n · log(value_range)). → NO: → Stage 5 — GREEDY + SEGMENT TREE / SORTED SET. Greedy choice requires range min/max on dynamic collection. Stage identification summary: Q3 YES (obvious sort key) → Stage 1/2. O(n log n). Q4 YES (dynamic heap) → Stage 3. O(n log n). Q5 YES (regret/undo) → Stage 4. O(n log n). Q6 YES (string comparator) → Stage 4. O(n log n). Q7 YES (binary search wrapper) → Stage 5. O(n log² n). If your solution is O(n²), look for the heap or sorted-set optimisation that makes each greedy step O(log n).
When This Technique BREAKS
🚩 [Stage 1] 0/1 Knapsack disguised as fractional knapsack — Greedy by value/weight ratio is optimal for fractional but provably WRONG for 0/1. Always check: are items divisible? If not, use 0/1 Knapsack DP.
🚩 [Stage 1] Activity selection with weighted intervals — Greedy (sort by end, take non-overlapping) maximises COUNT but not total WEIGHT. For weighted intervals, use Weighted Job Scheduling DP (sort by end + binary search + DP).
🚩 [Stage 2] Greedy on circular arrays without handling wrap-around — Circular problems (Gas Station, circular interval coverage) silently fail without wrap-around handling. Strategy: run the greedy twice on the doubled array, or use the "total feasibility + start-point" two-observation approach.
🚩 [Stage 3] Greedy on strings without the exchange-argument comparator — Alphabetical or length-based order for "largest number" or "minimum cost arrangement" gives wrong answers. Always derive the comparator from the exchange argument: a+b vs b+a.
🚩 [Stage 4] Regret greedy without the "virtual undo" mechanism — Problems like Min Refueling Stops require decisions that look non-greedy. Without the deferred-decision heap mechanism, the solution seems to require DP. Recognise: "collect all options passed; pick best when forced to decide."
🚩 [Stage 4] Exchange argument comparator that is not a total order — If a < b is not transitive or antisymmetric, std::sort / Python's sorted gives undefined behaviour. Always verify: key(a) < key(b) and key(b) < key(c) implies key(a) < key(c).
🚩 [Stage 5] Greedy + binary search: checking feasibility incorrectly — A bug in the inner greedy check produces a wrong binary-search boundary. Test the greedy check independently before wrapping in binary search.
Proof of Correctness Skeleton
Stage 1 — Activity Selection (Exchange Argument)
Claim: Sorting by end time and greedily keeping non-overlapping intervals maximises the count.
• Let G = greedy solution (sorted by end), O = optimal (same sort). Suppose they first differ at position k: G chose interval g, O chose o, with end(g) ≤ end(o).
• Swap o for g in O → O'. Since end(g) ≤ end(o), interval g ends no later than o. Therefore any interval compatible with o after k is also compatible with g. O' is valid and |O'| = |O|.
• Repeat for all differing positions. G transforms to O without reducing count. ∴ G is optimal. □
Stage 4 — Largest Number (String Comparator)
Claim: Sorting strings by the comparator a+b > b+a produces the largest concatenation.
• Consider any two adjacent strings a, b in the sorted order. If a+b ≥ b+a, then a comes before b.
• Swapping them produces ...b+a... whose concatenation is ≤ ...a+b.... So no swap improves the result — sorted order is globally optimal by the bubble-sort argument.
• Transitivity: must be verified; string concatenation comparison over the same alphabet IS transitive. □
Stage 4 — Regret Greedy: Minimum Refueling Stops
Claim: The "collect all passed stations in max-heap, refuel from best when empty" strategy minimises refueling stops.
• Invariant: At every moment, the heap holds exactly the stations we could have refueled at and are still available for retrospective use.
• Retrospectively choosing the highest-capacity passed station is equivalent to having made the optimal real-time choice at that station.
• Proof: Consider any optimal O that refuels at station i before our greedy does. Our greedy has the same tank level when reaching i (same route, same prior choices). If greedy later picks a higher-capacity station j > i, it has at least as much fuel at all subsequent points. □

Red Flags / Common Mistakes
🚩 [Stage 1] Applying greedy to 0/1 knapsack — Ratio-based greedy is a common instinct. It is ONLY correct when items are divisible or structure is a matroid. For 0/1 knapsack, two medium items beat one large. Correct model: "Greedy on ratios requires provable independence — verify via exchange argument or matroid."
🚩 [Stage 1] Forgetting that exchange argument requires TOTALITY — If the comparator is not transitive, sort() produces implementation-defined results. Correct model: "If a < b and b < c then a < c. Test with three examples before trusting your comparator."
🚩 [Stage 3] Not considering the regret mechanism — Min Refueling Stops and Course Schedule III LOOK like DP because "future info seems necessary." Regret greedy gives the optimal answer because the "best retrospective choice" equals the "best real-time choice." Correct model: "If the best retrospective and best real-time choice are the same element, regret greedy is correct."
🚩 [Stage 4] Left-to-right greedy on digit problems — Remove K Digits, Monotone Increasing Digits require right-to-left or a monotone stack because cascading effects propagate rightward. Correct model: "Digit problems with cascading effects need a stack (to undo recent choices) or right-to-left processing."
🚩 [Stage 4] Missing the input transformation — Jump Game variants, Min Taps, Talent Show require a non-trivial pre-processing step before greedy. Correct model: "First ask: what is the canonical form? Transform input, THEN apply greedy."
🚩 [Stage 2] Using < instead of <= for interval overlap — Balloon problems use <= (touching counts); meeting-room problems use <. One character determines correctness. Correct model: "Always check: does the problem treat touching endpoints as overlapping? Match the comparator precisely."
🚩 [Stage 3] Two-pass greedy: both passes in the same direction — For Candy, running both passes left-to-right misses the right-to-left constraint. Correct model: "Directional constraints require one pass in each direction; merge with max() to satisfy both simultaneously."
