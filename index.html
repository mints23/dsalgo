#01

Two Pointers

Technique T1

💡 Why This Topic Matters

Two pointers places one index at each end (or both starting from same side) and converges/diverges them based on a comparison. The core invariant is that the search space shrinks monotonically — once you move a pointer, you never need to revisit its previous position. It appears in Google interviews because it converts O(n²) brute force into O(n) for sorted-array pair/triplet problems, palindrome checks, in-place array mutations, and linked-list cycle detection. Mastery requires knowing WHICH pointer to move and WHY — the proof that moving the other pointer can never produce a better answer.

🔀 Sub-Variants to Master

  1. Opposite-direction on sorted array (two-sum, container, 3-sum)
  2. Same-direction fast/slow (cycle detection, middle of list, duplicate number)
  3. Shrink-from-both-ends palindrome / valid check
  4. Three pointers (Dutch National Flag, 3-way partition)
  5. Two pointers on two separate sorted arrays (merge, intersection)
  6. Two pointers with a running condition (boats, partitioning)
  7. Linked list two-pointer (Floyd's cycle, k-th from end, intersection)
  8. In-place array compaction (remove element, remove duplicates)
  9. Two-pointer binary search hybrid (find K closest elements)
  10. Two-pointer + greedy proof (trapping rain water)

  • "Longest/shortest subarray with ALL POSITIVE numbers" → sliding window (same-direction)
  • "Detect cycle in linked list" → fast/slow
  • "Find duplicate in array where values are in range [1,n]" → fast/slow implicit list
  • "Partition array in-place" → Dutch National Flag
  • "Pair heaviest with lightest" → sort + two ends

⛔ When This Technique BREAKS

  • Unsorted arrays where no monotone invariant exists
  • Negative numbers in subarray sum problems (monotonicity breaks)
  • Problems requiring ALL pairs not just existence
  • Non-linear search spaces (use binary search instead)
  • When you need index positions of all solutions (two-pointer counts but doesn't enumerate cheaply)

🚩 Red Flags / Common Mistakes

  • Moving the wrong pointer — always prove why the other pointer's remaining positions are dominated
  • Forgetting to skip duplicates in 3Sum/4Sum

  • Off-by-one in cycle entry (forgetting to reset one pointer to head)

  • Not handling edge case: array length < 2
#02

HashMap / HashSet

Data Structure T1

💡 Why This Topic Matters

A hash map reduces lookup/insert/delete to amortized O(1), converting quadratic brute-force scans into linear passes. The core invariant is that any problem requiring 'have I seen X before' or 'how many times did X appear' can be answered in O(1) with a map. It is ubiquitous in Google interviews because it is rarely the entire solution — it is the enabler that makes another technique (two-pointer, prefix sum, greedy, DP) efficient. Mastery means knowing when to use frequency map vs. existence set vs. value-to-index map vs. prefix-state map.

🔀 Sub-Variants to Master

  1. Frequency counting (anagrams, most frequent, mode)
  2. Existence / membership check (two-sum, contains duplicate)
  3. Value-to-index map (two-sum index return, next greater element)
  4. Grouping by canonical key (group anagrams, isomorphic strings)
  5. Prefix sum + hashmap (subarray sum = k, longest balanced)
  6. Complement lookup (two-sum, pair with given XOR)
  7. Sliding window with frequency map (minimum window substring)
  8. Two-pass: build map then query
  9. Nested map / map of sets (user activity minutes)
  10. Custom hash / polynomial rolling hash for strings

🎯 Pattern Recognition Triggers

  • "Have I seen X before?" → HashSet
  • "Count frequency of X" → HashMap
  • "Group items by property" → HashMap
  • "Find pair summing to target in O(n)" → complement map
  • "Longest subarray where [condition on prefix]" → prefix + earliest-index map
  • "Count subarrays where sum = k" → prefix frequency map

⛔ When This Technique BREAKS

  • When order of iteration matters and HashMap gives non-deterministic order (use LinkedHashMap)
  • When keys are floating-point (hash collisions; use string representation of reduced fractions)
  • When the map grows to O(n) and memory is the bottleneck
  • When you need range queries over keys (use TreeMap/sorted structure instead)
  • Multiset semantics needed (HashMap can't track 'remove one occurrence' without careful count management)

🚩 Red Flags / Common Mistakes

  • Using only one direction map for isomorphism (need bidirectional)
  • Forgetting (r%k+k)%k normalization for negative prefix remainders
  • Not seeding map with {0:1} for prefix sum problems

  • Not handling collision between 'zero occurrences' and 'key absent' (use getOrDefault)
  • Building O(n²) solution when map would give O(n)
#03

Binary Search

Algorithm T1

💡 Why This Topic Matters

Binary search eliminates half the search space per step, giving O(log n) on any problem with a monotone predicate. Its power beyond sorted arrays is the key insight: if you can define f(x) = true/false such that the answer space is [false...false, true...true], you can binary search on it regardless of whether it is an array, a value range, a time axis, or an answer space. Google interviewers specifically test this because 80% of candidates only know the 'find target in sorted array' template — the real skill is recognising the hidden monotone predicate.

🔀 Sub-Variants to Master

  1. Standard: find target in sorted array (lower/upper bound)
  2. Find first/last occurrence (leftmost/rightmost boundary)
  3. Search in rotated sorted array
  4. Search in 2D matrix
  5. Binary search on answer value (minimise maximum, maximise minimum)
  6. Binary search on time/capacity (feasibility function)
  7. Binary search on floating-point range
  8. Peak finding (bitonic array)
  9. Binary search on implicit sorted sequence
  10. Binary search + greedy/counting (k-th smallest, allocation problems)

🎯 Pattern Recognition Triggers

  • "Find in sorted array" → standard binary search
  • "Minimum X such that condition holds" → binary search on answer
  • "Maximum X such that condition holds" → binary search on answer (flip feasibility)
  • "K-th smallest in sorted matrix/pairs" → binary search on value + count
  • "Can we do this task in D days/units?" → feasibility function → binary search
  • "Floating point precision answer" → binary search on float range

⛔ When This Technique BREAKS

  • Non-monotone predicate (the search space must be strictly bipartitioned)
  • When the feasibility function itself is expensive (O(n log n)) making total O(n log² n) — may be too slow
  • Unsorted input where sorting would destroy index information
  • Problems requiring the exact position not just existence (binary search finds boundary, not arbitrary index)

🚩 Red Flags / Common Mistakes

  • Infinite loop from wrong mid formula: always use lo+(hi-lo)/2
  • Off-by-one in boundary: lo
  • Wrong feasibility direction (minimise max vs maximise min are mirror images)
  • Forgetting to handle when target not found (return -1 vs insert position)
  • Binary searching on unsorted or non-monotone space
  • Not verifying the feasibility function is correct before plugging into BS
  • Using floating point comparisons without epsilon (while hi-lo > 1e-9)
